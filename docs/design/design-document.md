## Section 1: Introduction & Overview

### What is being built

A portable, installable, git-tracked Claude Code configuration system that transforms the existing `~/.claude/` setup into a sharable repository with:

- **One-command installation** on any macOS/Linux machine
- **Autonomous CI/CD** that validates changes, reviews PRs, and proposes improvements
- **16 stack templates** (up from 3) covering web, mobile, specialized, and backend
- **Agent catalog** with 60+ agents (40 core + 20+ imported from community)
- **4-tier model routing** for optimal token efficiency
- **Preset agent teams** for common workflows

### Who it's for

- **Primary:** The author — portable setup across personal machines with version history
- **Secondary:** Other developers who want a production-grade Claude Code configuration
- **Tertiary:** Teams that want a shared, standardized AI-assisted development environment

### Key value proposition

Nobody else has a portable, self-improving Claude Code configuration. The closest competitors (wshobson/agents, everything-claude-code, oh-my-claudecode) provide agents and skills but not lifecycle hooks, stack templates, SDLC enforcement, autonomous pipelines, or Ghost Mode. This project packages the entire system as an installable, versionable, self-improving repository.

### Core concepts

| Concept | Definition |
|---------|-----------|
| **Base config** | Shared configuration tracked in git: commands, agents, hooks, rules, templates, skills |
| **User override** | Personal configuration NOT tracked: settings.local.json, personal CLAUDE.md additions, learning database |
| **Install script** | `install.sh` that symlinks base config to `~/.claude/` and sets up user override templates |
| **Stack template** | YAML file containing everything needed to scaffold a project: init commands, directories, starter files, CLAUDE.md, AGENTS.md |
| **Agent catalog** | JSON registry mapping agents to capabilities, model tiers, team memberships, and sources |
| **Autonomous CI** | GitHub Actions workflows that validate, version, and autonomously improve the configuration |

### 2.4 Settings Merge Strategy

```
settings.json (base, tracked)     settings.local.json (personal, gitignored)
┌──────────────────────────┐     ┌──────────────────────────────┐
│ permissions.allow: [...]  │     │ permissions.allow: [...]      │  ← MERGED (union)
│ permissions.deny: [...]   │     │ permissions.deny: [...]       │  ← MERGED (union)
│ hooks: {...}              │     │ hooks: {...}                  │  ← DEEP MERGED
│ env: {                    │     │ env: {                        │  ← MERGED (override)
│   AUTOCOMPACT: "65",      │     │   CUSTOM_VAR: "value",       │
│   SUBAGENT_MODEL: "sonnet"│     │   SUBAGENT_MODEL: "opus"     │  ← Local wins
│ }                         │     │ }                             │
│ enabledPlugins: {...}     │     │ enabledPlugins: {...}         │  ← MERGED (union)
│ statusLine: {...}         │     │ statusLine: {...}             │  ← Local wins if set
└──────────────────────────┘     └──────────────────────────────┘
              │                                │
              └────────────┬───────────────────┘
                           ▼
              ┌──────────────────────────┐
              │ Effective settings.json   │
              │ (Claude Code reads this)  │
              └──────────────────────────┘
```

**Note:** Claude Code natively supports `settings.local.json` as an overlay on `settings.json`. No custom merge logic is needed — Claude Code handles this internally.

### 3.3 Hook Contract

Hooks receive context via environment variables and stdin:

| Variable | Available In | Description |
|----------|-------------|-------------|
| `CLAUDE_FILE_PATH` | PostToolUse (Write/Edit) | Absolute path of the file just written/edited |
| `CLAUDE_TOOL_NAME` | PostToolUse, PreToolUse | Name of the tool being used |
| `CLAUDE_SESSION_ID` | All hooks | Current session identifier |
| `CLAUDE_PROJECT_DIR` | All hooks | Working directory of the project |

**Exit codes:**
- `0` — success, proceed normally
- `1` — error, but don't block (logged as warning)
- `2` — block the action (PreToolUse only)

**Prompt hooks** return JSON: `{"decision": "allow"|"block"|"ask", "reason": "string"}`

## Section 4: Implementation Details

### 4.1 install.sh Design

```
Usage: install.sh [OPTIONS]

Options:
  --mode=symlink|copy    Install mode (default: symlink)
  --dry-run              Show what would be changed without doing it
  --no-backup            Skip backing up existing ~/.claude/
  --prefix=PATH          Install to PATH instead of ~/.claude/
  --modules=LIST         Comma-separated: core,agents,templates,hooks,rules,skills (default: all)
  --help                 Show this help

Steps:
  1. Pre-flight: check git, bash 4+, jq
  2. Clone or update repo at ~/.claude-super-setup/
  3. If ~/.claude/ exists and --no-backup not set: backup to ~/.claude-backup-{YYYYMMDD-HHMMSS}/
  4. Create ~/.claude/ if needed
  5. For each module in --modules:
     - In symlink mode: create symlinks from ~/.claude/{module} → repo/{module}
     - In copy mode: rsync repo/{module}/ → ~/.claude/{module}/
  6. Copy config/settings.json → ~/.claude/settings.json (if not exists, or merge)
  7. Copy config/CLAUDE.md → ~/.claude/CLAUDE.md (if not exists, or merge)
  8. Create user override files from templates (if not exist):
     - ~/.claude/settings.local.json from user-overrides/settings.local.json.template
     - Personal CLAUDE.md additions notice
  9. chmod +x hooks/*.sh
  10. Run health check (validate all symlinks, parse settings.json, count files)
  11. Print summary: installed modules, agent count, template count, hook count
```

### 4.4 Agent Import Process

When importing agents from external ecosystems (VoltAgent, wshobson, senaiverse):

1. **Identify candidates:** Review external agent catalog, select agents that fill capability gaps
2. **Adapt front-matter:** Add required fields (department, model_tier, capabilities, teams) to match our catalog schema
3. **Normalize tools list:** Map external tool names to our tool names
4. **Adjust model assignment:** Assign to appropriate tier (most community agents → sonnet)
5. **Place in community/:** File in appropriate subdirectory (language-specialists/, data-ai/, infrastructure/, mobile/)
6. **Register in catalog.json:** Add entry with `source: "community"` and `source_repo` reference
7. **Add attribution:** Note original repo in agent file header
8. **Validate:** Run CI to ensure front-matter validates

## Section 5: Conventions & Design Patterns

### 5.1 File Naming

| File Type | Convention | Example |
|-----------|-----------|---------|
| Commands | kebab-case.md | `auto-build-all.md` |
| Agents | kebab-case.md | `security-auditor.md` |
| Hooks | kebab-case.sh | `auto-fix-loop.sh` |

| Stack templates | category-name.yaml | `web-astro.yaml`, `mobile-flutter.yaml` |
| Schemas | kebab-case.schema.json | `stack-template.schema.json` |

### 5.2 Agent Department Taxonomy

Maintain existing 8 departments. A new department is only created when 3+ agents share a clear domain that doesn't fit existing departments.

| Department | Purpose | Core Agents | Community Agents |
|------------|---------|-------------|-----------------|
| engineering | Code implementation | architect, backend-dev, frontend-dev, mobile-app-builder | language specialists, infrastructure |
| testing | Quality assurance | tdd-test-writer, test-writer-fixer, visual-tester | mobile testing specialists |
| product | Product management | sprint-prioritizer, feedback-synthesizer | — |
| design | UI/UX | ui-designer, whimsy-injector | — |
| marketing | Growth & content | tiktok-strategist, growth-hacker, content-creator | — |
| studio-operations | Process & coordination | studio-coach, studio-producer, orchestrator | — |
| project-management | Planning & shipping | project-shipper, experiment-tracker | — |
| bonus | Miscellaneous | joker | — |

### 5.3 Versioning Strategy

Semantic versioning: `MAJOR.MINOR.PATCH`
- **MAJOR:** Breaking change to install.sh, settings.json schema, or hook contracts
- **MINOR:** New agents, new stack templates, new commands, new rules
- **PATCH:** Bug fixes to existing hooks, command wording, typos, template dependency updates

Release automation: conventional commit messages → release-please → CHANGELOG → GitHub Release.

### 5.4 Template Variable System

Stack templates use literal values, not variables. The `/new-app` command reads the YAML and executes the `init_commands` in a new directory. The only runtime substitution is the project directory name, which is handled by the scaffolding command — not by the template.
